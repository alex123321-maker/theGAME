# Бэклог перехода процедурной карты в изометрию и добавления runtime regeneration

## Назначение
Этот backlog задаёт конкретный порядок работ для ИИ-агента, который должен перевести текущий foundation-проект процедурной карты на изометрическую подачу, подключить минимальный набор ассетов и сделать удобную перегенерацию карты прямо во время работы программы.

Основание для backlog: игровое описание проекта и требования к процедурной карте деревни. Источник истины по карте и архитектуре — `MapData`, `TileData`, генерационные passes и валидация.

---

## Целевое состояние
К завершению этой инициативы проект должен уметь:

1. Показывать локальную карту в изометрии.
2. Использовать минимальный набор рабочих ассетов вместо абстрактного прямоугольного greybox.
3. Перегенерировать карту в рантайме тремя способами:
   - `Regenerate Current` — заново строит карту по текущему seed.
   - `Randomize Seed` — генерирует новый seed и строит новую карту.
   - `Apply Seed` — применяет вручную введённый seed.
4. Показывать разработчику seed, статус валидации, активный overlay и краткую диагностическую сводку.
5. Не ломать существующий data-driven контракт карты.

---

## Инварианты, которые агент не имеет права ломать

1. `MapData` остаётся источником истины для логики карты, а не `TileMapLayer` и не `TileSet`.
2. Генератор остаётся детерминированным по `seed` и версии генератора.
3. Изометрия — это слой представления, а не причина переписывать генератор.
4. Debug overlays обязаны пережить переход на красивые ассеты.
5. Runtime regeneration работает без перезапуска приложения.
6. Все изменения должны быть изолированными: не смешивать визуальную миграцию и рефактор всего проекта в одном шаге.
7. Любой новый presentation-only слой не меняет `walk_cost`, `buildable`, `future_wallable`, `threat_value` и другие логические поля, если это явно не записано в `MapData`.

## Минимальный набор ассетов для первой изометрической версии

### Terrain base
- `ground`
- `clearing`
- `road`

Минимум: по 3–4 варианта каждого.

### Boundary / obstacle tiles
- `forest edge`
- `rock edge`
- `water edge`
- `ravine edge`

Минимум: по 2–3 варианта каждого.

### Transition tiles
- `road_to_ground`
- `clearing_to_ground`
- `obstacle_transition`

Минимум: по 2 варианта.

### Props
- `boulder`
- `stump`
- `dead_tree`
- `ruins_fragment`

Минимум: 4–6 спрайтов.

### Debug overlays
- marker точек входа
- buildable mask
- threat heat
- height tint

### Стиль ассетов
- минималистичный ручной скетч
- монохромная чернильная/карандашная графика
- светлый почти белый фон
- строгая прямоугольная изометрия
- низкий визуальный шум
- дорожка, лес, камень, вода и овраг должны считываться с первого взгляда

# Backlog задач

## ISO-01 — Зафиксировать визуальный контракт изометрии
**Цель:**
Отделить логическую сетку от экранной проекции так, чтобы генератор вообще не знал об изометрии.

**Что меняется:**
- Добавить единый модуль `world_projection.gd`.
- Зафиксировать функции:
  - `logical_to_screen()`
  - `screen_to_logical_approx()`
  - `get_tile_anchor_offset()`
- Зафиксировать размер изометрической клетки, origin и способ якорения тайлов.

**Где менять:**
- `scripts/presentation/world_projection.gd`
- `docs/IMPLEMENTATION_NOTES.md`
- при необходимости `scripts/core/map/map_types.gd`

**Зависимости:** нет.

**Критерий готовности:**
Любой логический тайл можно стабильно перевести в экранную позицию, а `map_renderer` больше не содержит зашитые формулы проекции.

**Запрещено:**
- вставлять формулы изометрии прямо в генератор
- менять сериализацию карты ради рендера

**Цель:**

**Что меняется:**

  - randomize new seed

- Сохранять состояние камеры, grid и overlay mode.
- При перегенерации очищать только визуальные узлы карты и собирать их заново по новым данным.

**Где менять:**

**Критерий готовности:**

**Запрещено:**

# Definition of Done для всей инициативы

Инициатива считается завершённой, если одновременно выполнены все условия:

1. Карта визуально изометрическая и читается лучше исходного прямоугольного greybox.
2. Все базовые `terrain_type` отображаются собственными тайлами, а не цветовыми заглушками.
3. Во время работы программы доступны `Regenerate Current`, `Randomize Seed`, `Apply Seed`.
4. После перегенерации не остаётся визуальных артефактов от прошлой карты.
5. Не меньше 10 reference seed проходят генерацию и валидацию подряд.
6. Документация проекта обновлена под новый изометрический и runtime workflow.

# Короткий шаблон постановки задач агенту

Формат:

`Цель -> ограничения -> какие файлы менять -> какой артефакт получить -> критерий завершения -> какие инварианты нельзя ломать`

### Пример 1
Переведи `map_renderer` на `TileMapLayer`-стек в изометрии. Не трогай `MapData` и генератор. Измени только `scenes/map_view/*` и `scripts/presentation/*`. Результат: карта рисуется через `TileSet`. Завершение: проходят 3 фиксированных seed, overlay-слои не пропали.

### Пример 2
Добавь runtime-панель `seed/debug`. Не трогай генерационные `passes`. Результат: можно ввести seed и нажать `Apply` без перезапуска. Завершение: камера сохраняется, old nodes очищаются, validation status обновляется.
